## Original Post

## Idea

When Crunch picks up an issue but another run for the same issue is already in progress, there's no visible signal to Marcus that the issue has been seen.

**Current behavior**: per-issue concurrency groups silently queue the second run in GitHub Actions — nothing shows on the issue itself.

**Proposal**: Add an instant pre-flight job (no concurrency constraint) that fires immediately and posts a comment or label the moment a trigger fires — before the concurrency lock kicks in.

Example comment:
> 😴 Issue queued — another run for this issue is in progress. Will pick this up next.

## Implementation sketch

Split agent.yml into 3 jobs:
- `signal` job: instant, no concurrency → adds `crunch/queued` label or posts ack comment  
- `work` job: per-issue concurrency, needs: signal → heavy LLM work  
- `write` job: global concurrency queue, needs: work → serialized git push  

Note: GitHub's reaction API only supports 8 emojis — no actual 😴. A **label** like `crunch/queued` is the cleanest visible marker.

## Trade-offs
- ~30s overhead per run (extra job spin-up)
- Clear visibility on the issue list that Crunch has seen it
- Queue behavior stays identical
